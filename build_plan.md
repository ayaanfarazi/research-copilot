# AI Investment Research Copilot — Build Spec (v7)

**Scope:** US public companies only. **Builder:** limited coding experience, using Claude Code + Cursor. **Budget:** ~62h core + ~8h slack.

**One sentence:** Enter a US ticker → the tool turns that company's SEC filings into a structured, fully-cited deal-readiness brief centered on *whether the company can survive and service its capital structure* — all financials computed deterministically in code (never generated by the LLM), every figure traceable to its XBRL tag/period/filing, plus an AI reasoning layer that takes a defensible view, argues both sides of management's EBITDA add-backs, and flags where the MD&A narrative diverges from the numbers.

---

## 1. The two ideas the whole project rests on

**(a) The number boundary.** The LLM never *produces or retrieves* a number — but it freely *reasons over* numbers the deterministic layer hands it. Generating a figure is forbidden (hallucination risk, untraceable); synthesizing handed-in figures ("net leverage of 2.1x, down from 3.4x, is strong deleveraging") is the entire point. The architecture isn't "keep the AI away from numbers," it's "keep the AI from being the *source* of numbers." This is the single most important interview answer to "isn't this a wrapper?"

**(b) The spine.** The headline question is *"can this company carry, service, and refinance its debt — and are the earnings real?"* That question is what Rx, LevFin, and PE all run on, and it maps to the ADCB experience (DSCR, leverage, tranching, cash waterfalls). Operating metrics (margins, growth, ROE) appear as a supporting panel — free off the same data — but don't define the product. M&A is explicitly a **separate, later** tool (§13), not bolted on here.

---

## 2. Non-negotiables (the anti-wrapper core)

1. LLM never produces/retrieves a number; it receives computed figures by ID and references them by ID.
2. All financials, ratios, and the risk scorecard are computed deterministically in code from XBRL.
3. Every number expands to: filing, period, XBRL tag(s), formula.
4. Every qualitative/reasoning claim cites a filing section or a computed figure-ID. An unanchored claim = validation failure.
5. Output validation: scan the assembled brief for any numeric token the model emitted that isn't in the allowed data-layer set; if found, fail. Applies to the reasoning panels exactly as to descriptive ones.
6. The AI may *characterize* (strong/thin/fragile) and *weigh* (which risk binds) freely — but on top of a cited figure, never replacing one.
7. Never silently fail: every figure shows its resolved tag + confidence tier, or explicitly shows "not found." Zero/blank are not acceptable.

## 5. Data spine — SEC EDGAR

Free, no API key. **Required:** descriptive User-Agent header (e.g. `Ayaan Farazi research-copilot ayaan@email.com`) or EDGAR blocks you. Rate limit 10 req/sec — throttle.

- Ticker→CIK: `https://www.sec.gov/files/company_tickers.json`
- Filing history: `https://data.sec.gov/submissions/CIK{10-digit-zero-padded}.json`
- All XBRL facts: `https://data.sec.gov/api/xbrl/companyfacts/CIK{10-digit}.json`
- Single concept series: `https://data.sec.gov/api/xbrl/companyconcept/CIK{10-digit}/us-gaap/{Tag}.json`
- Tag frequency across all filers: `https://data.sec.gov/api/xbrl/frames/` (used to build the resolver, §6)
- 10-K text: get accession # + primary doc from submissions API, then fetch the filing HTML from EDGAR archives.

**The three hard data problems:** (i) *tag variation* — same concept, different tags across companies; (ii) *deduplication* — companyfacts returns every fact for a concept across every filing (amendments, restatements, prior-year comparatives), so you must select one clean value per period (filter by `form`, latest `filed`, `fy`/`fp`); (iii) *annual isolation* — use `form`=10-K + frame/period to get clean FY values.

## 6. Tag resolution — works honestly for (almost) any company

The principle: for any concept on any company, the tool either resolves to a value **and shows which tag + confidence tier**, or shows **"not found."** Never zero, never silence, never a wrong tag unlabeled.

**Component 1 — frequency-ranked candidate lists (built empirically).** For each concept, hold a priority-ordered list of tag variants ranked by how many filers use each (pull this from the `/frames/` endpoint — ~2-3h one-time). Resolver tries them in order, takes the first that returns data. Starting lists (extend empirically):

| Concept | Priority order |
|---|---|
| Revenue | `RevenueFromContractWithCustomerExcludingAssessedTax` → `Revenues` → `SalesRevenueNet` → `SalesRevenueGoodsNet` |
| COGS | `CostOfGoodsAndServicesSold` → `CostOfRevenue` → `CostOfGoodsSold` → `CostOfSales` |
| Operating income | `OperatingIncomeLoss` |
| Net income | `NetIncomeLoss` |
| Interest expense | `InterestExpense` → `InterestAndDebtExpense` → `InterestExpenseDebt` |
| D&A | `DepreciationDepletionAndAmortization` → `DepreciationAndAmortization` → `DepreciationAmortizationAndAccretionNet` |
| Operating cash flow | `NetCashProvidedByUsedInOperatingActivities` |
| Capex | `PaymentsToAcquirePropertyPlantAndEquipment` → `PaymentsToAcquireProductiveAssets` |
| Current debt | `DebtCurrent` → `LongTermDebtCurrent` → `ShortTermBorrowings` |
| Non-current debt | `LongTermDebtNoncurrent` → `LongTermDebt` |
| Cash | `CashAndCashEquivalentsAtCarryingValue` → `CashAndCashEquivalents` |
| ST investments | `ShortTermInvestments` |
| Total assets / liabilities / equity | `Assets` / `Liabilities` / `StockholdersEquity` (→ `...IncludingPortionAttributableToNoncontrollingInterest`) |
| Op lease liabilities | `OperatingLeaseLiabilityCurrent`, `OperatingLeaseLiabilityNoncurrent` |
| Shares out / diluted shares / diluted EPS | `dei:EntityCommonStockSharesOutstanding` / `WeightedAverageNumberOfDilutedSharesOutstanding` / `EarningsPerShareDiluted` |

**Component 2 — resolver returns a structured object, never a bare number:**
```
for tag in candidates[concept]:
    facts = companyfacts[cik]['facts']['us-gaap'].get(tag)
    if facts:
        v = deduplicate_and_select(facts, form='10-K', fiscal_year=year)
        if v is not None:
            return ResolvedFact(value=v, tag=tag, confidence=tier_for(tag))
return ResolvedFact(value=None, tag=None, confidence='not_found')
```

**Component 3 — confidence tier on every figure:** *Verified* (one of the 5 hand-verified demo cos, checked vs the 10-K) → green; *High* (top-3 frequency tag) → default, no badge; *Low* (rank 4+) → amber "check source"; *Not found* → red "see filing."

**Component 4 — graceful degradation:** a "not found" input cascades to "not found" on derived metrics (e.g. no interest expense → interest coverage = "not found," not a divide-by-zero). Tearsheet always renders; gaps show honestly.

## 7. Metric spec

Each rendered figure stores **value, period, source tag(s), formula.**

**Constructed inputs (with judgment calls — document each):**
- **EBITDA** = OperatingIncomeLoss + D&A (cash-flow). D&A tag is messy (may bundle intangible amort) — render EBITDA as an explicit reconciliation (op income → +D&A → EBITDA, each component tagged), not a single number. Alt build: NI + interest + taxes + D&A.
- **Total debt** = current debt + non-current debt. Default exclude operating leases; flag the choice; offer a "with leases" toggle as a nicety.
- **Net debt** = total debt − cash − ST investments.

**Operating panel (Investor view):** gross/operating/net margin; revenue YoY growth; revenue CAGR `(end/start)^(1/n)−1`; ROE = NI ÷ avg equity (state ending vs avg); ROE DuPont = (NI/Rev)(Rev/Assets)(Assets/Equity); FCF margin.

**Credit panel (Credit view):** total leverage (debt/EBITDA); net leverage (net debt/EBITDA); interest coverage (EBITDA/interest); cash interest coverage ((EBITDA−capex)/interest); FCF (OCF−capex); FCF/total debt; liquidity (cash + ST inv + revolver if found). *Skip true FCCR* — needs mandatory amortization, not cleanly extractable; don't fake precision.

**Valuation panel (stretch, needs external price — label it external everywhere):** market cap (shares×price); EV (mktcap + debt − cash + minority + preferred); EV/EBITDA, EV/EBIT, EV/Revenue, P/E. *Comps belong to the §13 M&A tool — keep this minimal.*

**Covenant-style screen:** flag leverage/coverage vs *illustrative* bands (e.g. total leverage 4-6×; interest coverage 2-3× floor) → green/amber/red + the actual number; plus the net-leverage trajectory. **Hard rule:** label these *illustrative screening bands, not real covenants* (real covenants live in the credit agreement, not the 10-K). ~1-2h off data you already have.

**Survival panel (the spine — core, not stretch):**
- *Maturity wall* — per-tranche principal + maturity year from the debt footnote (LLM extraction), cross-checked against XBRL total debt. **Graceful degrade:** if footnote extraction fights you, fall back to the XBRL current/non-current split.
- *Liquidity runway* — (cash + ST inv + revolver) vs near-term maturities and cash burn.
- *Deleveraging trajectory* — net leverage across 3-5 yrs (improving/flat/worsening).
- *Coverage durability* — coverage trend; flag thinning coverage even when leverage looks flat.

**Credit risk scorecard (deterministic — the anchor for the AI view):** assign a tier per dimension, roll up to a band via a *visible, documented* rule (e.g. weakest dimension dominates, or a point system). Keep it explainable — the point isn't a sophisticated model, it's an auditable verdict skeleton the AI then articulates.

| Dimension | Input | Illustrative tiers |
|---|---|---|
| Leverage | net leverage | ≤2 strong / 2-4 moderate / 4-6 stretched / >6 distressed |
| Coverage | interest coverage | >6 strong / 3-6 adequate / 1.5-3 thin / <1.5 critical |
| Trajectory | net-lev trend | improving / flat / worsening |
| Liquidity | runway vs near-term maturities | comfortable / adequate / tight / acute |

## 8. Document layer

Fetch 10-K HTML; split into Item 1 / Item 1A / Item 7 + debt footnote, keeping a section reference for each. **Don't build a general parser** — good-enough for the 5 demo companies. Watch out: "Item 1A" also appears in the table of contents, so naive regex grabs the wrong anchor. Consider feeding larger chunks with section hints and letting the model self-locate (you're citing sections anyway).

## 9. The reasoning + qualitative layer

**Prompt + validation rules (apply to ALL panels below):** model receives the text chunk + real numbers each tagged with an ID; references figures by ID only, never writes a raw number; every claim returns a `citation` (section or figure-ID); structured JSON against a fixed schema (use structured outputs / tool use); post-validation scans for stray numeric tokens and fails on any number outside the allowed set.

**Descriptive panels:**
- *Business summary* (Item 1), *company-specific risks* (Item 1A, separate boilerplate from specific), *revenue drivers* (Item 7 + segments + real numbers).
- *QoE bridge:* deterministic part pulls XBRL-tagged add-backs in code (`ShareBasedCompensation`, `RestructuringCharges`, `AssetImpairmentCharges`, `GoodwillImpairmentLoss`); LLM part surfaces *additional* claimed one-time items (litigation, M&A costs) as cited candidates **with no number of their own**. Render the bridge explicitly: GAAP EBITDA → +SBC → +restructuring → +impairments → +[flagged candidates] → management-style adjusted EBITDA, showing the resulting shift in adjusted leverage. Analyst toggles candidates on/off.

**Reasoning panels (the differentiation — all pure synthesis over deterministic inputs):**

- **(A) The view — anchored synthesis (headline).** Input: the scorecard + all computed figures (by ID) + relevant text. Output: a verdict that *explains the scorecard's band* — which dimension binds, why, what would change it — with every load-bearing clause tagged to a figure-ID. The band (skeleton) comes from code; the AI supplies articulation. Shape: *"Capital structure looks fragile [band from scorecard]: net leverage of {fig:net_lev} sits in the stretched tier, and while coverage of {fig:int_cov} is adequate, {fig:near_term_maturities} of the {fig:total_debt} stack matures within 18 months against {fig:liquidity} — refinancing is the binding constraint."* Run numeric validation on the output.

- **(B) Add-back adversary (headline).** For each add-back in the QoE bridge, argue both sides, cited: *bull* (management's justification, cited to filing) and *skeptic* (credit-committee challenge — recurring-in-disguise, suspiciously timed, large vs EBITDA, referencing the tagged magnitude by ID). The AI doesn't decide; it arms the analyst. Numbers stay deterministic; only argumentation is AI.

- **(C) Narrative-vs-number tension detector (headline, hardest).** Cross-reference MD&A/segment narrative vs the corresponding figures. Output flagged divergences, each naming the claim (cited) AND the contradicting figure (by ID). **Failure mode to design against:** false positives poison the whole tool's credibility. Mitigation — require both a citation and a figure-ID per flag; tune to **under-flag, not over-flag** (a missed tension disappoints; a fabricated one disqualifies); hand-check on all 5 demo companies. If precision can't be tamed in budget: ship labeled "experimental" or cut.

## 11. Build roadmap (~62h core + ~8h slack)

Module boundaries matter: keep SEC ingestion, XBRL resolution/dedup, grounding/citation, and provenance UI as clean separate modules — the §13 M&A tool reuses them wholesale.

**Phase 0 — Plumbing (~5h).** Python + Cursor/Claude Code + GitHub repo; pull companyfacts for one ticker and print it; one successful Anthropic API call. *Done when:* you can fetch SEC data and call Claude locally.

**Phase 1 — Data spine (~16h listed; plan for ~24).**
- Ticker→CIK; pull companyfacts; extract line items 3-5 yrs.
- Run the `/frames/` frequency pull (~2-3h) → build the priority-ordered resolver returning `ResolvedFact` (never a bare number; explicit `not_found`).
- Attach confidence tiers; implement graceful degradation.
- Dedup (latest filing per period) + annual isolation.
- Construct EBITDA / total debt / net debt; render the EBITDA reconciliation.
- Compute operating + credit ratios; build covenant screen, the XBRL-only survival pieces (runway, trajectory, coverage durability), and the **risk scorecard** (the view's anchor — must exist before Phase 2.5).
- **Hand-verify against the actual 10-K for 2 demo companies — non-negotiable.**
- *Done when:* demo tickers give clean verified output; arbitrary tickers give tagged+tiered values or explicit "not found"; scorecard produces an explainable band.
- ⚠️ **This phase overruns** (frequency pull + resolver add ~3-4h). If behind, the sacrifice is the Phase 4 stretch — never the hand-verification or the "not found" discipline.

**Phase 2 — Descriptive LLM layer + tranche extraction (~13h).**
- Split 10-K into Items 1/1A/7 + debt footnote (good-enough for 5 cos).
- Grounded prompts → structured JSON: business summary, boilerplate-stripped risks, revenue drivers, QoE bridge + candidate add-backs (each cited, figures by ID).
- Extract debt tranches + maturities for the maturity wall; cross-check sum vs XBRL total debt (degrade to current/non-current split if it overruns).
- **Get the numeric-token validation airtight here** — Phase 2.5 leans on it.
- *Done when:* descriptive sections grounded/cited/company-specific; bridge assembles; maturity wall renders; validation reliably catches stray numbers.

**Phase 2.5 — AI reasoning layer A/B/C (~10h).** Build in order of increasing risk:
- **(B) first** — most contained; reuses the bridge.
- **(A) second** — feed scorecard + figures; verdict explains the band, every clause tagged; validate output.
- **(C) last** — riskiest; require citation+figure-ID per flag, tune to under-flag, hand-check all 5 cos.
- *Done when:* all three produce anchored/cited/validated output on the 5 cos and C isn't fabricating divergences.
- ⚠️ **If tight, cut C first, then B — never A.** A is the headline and most defensible.

**Phase 3 — Assemble brief + UI (~10h).** Streamlit: ticker + view toggle → pipeline → tearsheet. Expand-to-source on every number *and every reasoning claim*. Render scorecard, survival panel, covenant screen, QoE bridge, and A/B/C as distinct visual blocks (demo money-shots). Polish the 5-company demo path. *Done when:* live end-to-end demo in both views, looks clean.

**Phase 4 — One stretch only (~8h, if budget remains).** Pick one: *refinancing-scenario toggle* (assume a near-term maturity refinances at a stressed rate, show the coverage/FCF hit — most spine-aligned, pure deterministic math); *"with leases" leverage toggle*; *minimal comps teaser* (2-3 peers, external price — but this really belongs to §13).
**NOT in scope:** DCF, Monte Carlo, full PE diligence module, multi-user/production hardening, banks/insurers (XBRL nightmare), a full comps engine (that's §13 — don't start it here).

## 13. The competitive framing + the deferred projects

**What this is NOT:** not a better/broader data source than Bloomberg/CapIQ/FactSet — don't compete on data. Provenance/"you can verify it" is a *recruiting* signal (shows you understand why LLM-generated numbers are dangerous), **not** a product moat (a no-name tool has *less* inherent trust). Not a defensible startup in public-filings form — public data is the least valuable info in finance because it's everywhere.

**What it genuinely IS:** it does analyst-reasoning tasks a database doesn't (take a view, argue add-backs, flag tension) in a way a generic AI wrapper can't do safely (every claim anchored to an independently-computed number the model can't fabricate). One-line pitch: *"AlphaSense finds you the passage; this tool computes the number independently and flags when the passage and the number disagree."* Honest ceiling: differentiated *today*, but a funded competitor could build similar in months — fine, since this is a recruiting artifact + learning engine, not a race.

**Recruiting honesty:** strong for Rx/LevFin/PE, A-tier for credit HF / fundamental AM, weak for S&T/consulting. It won't *get* you the offer — networking, technicals, GPA, and the ADCB/Mashreq story do that. It makes you *talkable* in those rooms (the maturity wall, the EBITDA bridge, and the AI reasoning layer let interviewers probe both deal judgment and technical/AI judgment). Don't let it eat the hours that belong to networking and technicals.

**Deferred project — M&A / valuation tool (§ future, do NOT build now).** M&A's native question is relative/transactional (comps, precedents, accretion/dilution) — multi-company, a different artifact. Build it *after* this one because it reuses this tool's entire data + grounding + provenance spine; the new work is peer selection, the comps engine, relative-valuation math. Narrative unlocked: "built a credit/survival tool, then extended the same grounded-data architecture into an M&A engine" = a platform + scoping judgment. **Guardrail:** no comps/peer engine in *this* repo. If you start building one here, stop.

**Deferred direction — private documents (§ long-term, do NOT build now).** The architecture's real value is pointing the same engine at private deal docs where no incumbent exists and the work is done by hand at 2am: CIMs, lender presentations, credit agreements (the *actual* covenants), management models. Progression: v1 ticker→tearsheet (this summer) → v2 upload a deal doc → extract leverage/covenants/maturities/add-backs with the same anchoring → v3 deal-room diligence assistant. The QoE add-back adversary is the seed of this. Stays deferred: needs private docs (no free EDGAR equivalent), a different parsing problem, and it's a *company*, not a summer project. Revisit only after you've sat in a credit/PE seat and felt the pain you'd be solving.
